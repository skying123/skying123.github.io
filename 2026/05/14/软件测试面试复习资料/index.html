一、软件测试基础理论

1.1 测试基础概念

测试的六条基本法则

  • 功能：软件是否满足需求规格
  • 可靠性：软件在规定条件下维持性能水平的能力
  • 易用性：用户学习、操作、理解软件的难易程度
  • 效率：软件执行功能时的资源消耗（时间、空间）
  • 可维护性：软件被修改的难易程度
  • 可移植性：软件从一个环境迁移到另一个环境的能力

软件测试分类（按是否关心内部结构）

类型 定义 特点
白盒测试 基于代码内部逻辑结构 需要懂代码，测试覆盖率高
黑盒测试 不考虑内部结构，只测功能 从用户角度出发
灰盒测试 介于白盒和黑盒之间 部分了解内部结构

测试阶段划分

黑盒测试包含内容

  • UI测试、配置测试、表单值域测试
  • 数据完整性测试、逻辑测试、业务流程测试
  • 逆向思维测试、接口测试、本地化测试
  • Cookie测试、易用性测试、错误推断测试
  • 安全性测试、兼容性测试、稳定性测试
  • 负载测试、压力测试、故障转移和恢复测试

1.2 测试用例设计方法

常用设计方法

  1. 等价类划分：将输入数据划分为若干等价类，每类选一个代表测试
  2. 边界值分析：重点测试边界条件（如0、最大值、最小值）
  3. 因果图法：分析输入条件之间的因果关系
  4. 场景法：基于业务流程设计测试场景
  5. 正交实验法：多因素多水平组合测试
  6. 错误推断法：基于经验推测可能的错误

编写测试用例原则

1.3 经典面试题

Q1: 如果能够执行完美的黑盒测试，还需要进行白盒测试吗？

答：需要。黑盒测试只能验证功能是否符合需求，无法发现代码层面的逻辑错误、资源泄露、性能瓶颈等问题。白盒测试可以检查代码覆盖率，发现死代码、冗余代码，确保代码质量。

Q2: 如何确定一个问题是不是BUG？

答：对照需求规格说明书，不满足需求就是BUG。如果需求不明确，需要与产品经理确认预期行为。

Q3: 产品测试到什么时候就算是足够了？

答：没有绝对的”足够”，通常满足以下条件可认为达到发布标准：

  • 所有P0/P1级用例执行通过
  • 严重缺陷（Critical/High）修复率100%
  • 中低优先级缺陷有明确规避方案或延期计划
  • 性能指标达到预期
  • 核心业务场景稳定运行一段时间

Q4: 优秀的测试工程师应该具备哪些素质？

  1. 业务理解能力：深入理解被测系统的业务逻辑
  2. 技术能力：掌握测试工具、编程语言、数据库等
  3. 分析能力：能快速定位问题根因
  4. 沟通能力：与开发、产品有效沟通
  5. 细心和耐心：测试工作重复性强，需要细致
  6. 学习能力：技术更新快，需要持续学习

五、接口测试

5.1 接口测试基础

什么是接口测试？

接口测试是测试系统组件间接口的一种测试，主要用于检测外部系统与系统之间以及内部各个子系统之间的交互点。

接口测试的重点

  1. 检查接口参数传递的正确性
  2. 检查接口功能实现
  3. 检查接口的异常处理能力
  4. 检查接口的性能和安全性

POST 创建资源 否
PUT 更新资源（全量） 是
PATCH 更新资源（部分） 否

六、性能测试

6.1 性能测试基础概念

常见性能测试类型

类型 定义 目的
负载测试 逐步增加负载，测试系统性能变化 确定系统最大承载能力
压力测试 超过正常负载，测试系统极限 发现系统瓶颈和崩溃点
并发测试 多用户同时操作 发现并发问题（死锁、资源竞争）
稳定性测试 长时间运行测试 发现内存泄露等问题

关键性能指标

  • TPS/QPS：每秒事务/查询数
  • 响应时间：从请求到响应的时间（平均、最大、90%线）
  • 并发用户数：同时在线操作的用户数
  • 吞吐量：单位时间处理的数据量
  • 资源利用率：CPU、内存、磁盘、网络使用率

6.2 Locust 性能测试（Python）

6.3 JMeter 关键概念（Java方向）

核心组件

  1. 线程组：模拟用户组，设置并发数、启动时间、循环次数
  2. 取样器：发送请求（HTTP、FTP、JDBC等）
  3. 断言：验证响应结果
  4. 监听器：收集和展示结果（聚合报告、图形结果）
  5. 配置元件：请求默认值、CSV数据配置、HTTP Cookie管理器
  6. 前置/后置处理器：请求前准备、响应后处理

集合点（Synchronizing Timer）

  • 作用：让多个用户在同一时刻发送请求，模拟真实并发场景
  • 使用场景：秒杀、抢购等瞬时高并发场景

参数化方式

  1. CSV Data Set Config：从CSV文件读取数据
  2. 用户定义的变量：全局变量
  3. 函数助手：随机数、UUID、时间戳等
  4. 正则表达式提取器：从响应中提取数据供后续使用

8.2 项目介绍话术

场景：介绍自动化框架

“我设计的框架采用分层架构，主要分为五层：

  1. API层：封装所有接口请求，按业务模块划分
  2. 测试用例层：只关注测试逻辑，调用API层方法
  3. 公共工具层：封装请求发送、日志记录、断言验证
  4. 配置层：YAML文件管理多环境配置
  5. 数据层：测试数据独立管理

这样设计的好处是职责清晰、易于维护，
修改接口只需改API层，修改数据不影响用例代码。”

场景：如何处理接口依赖

“接口依赖主要有两类：

  1. Token依赖：用fixture实现，在conftest.py中创建login_token的fixture，
    需要认证的用例直接引用
  2. 数据依赖：比如测试订单详情需要先创建订单，
    也用fixture实现，用yield保证数据用完后清理

复杂场景下，我会在API类中封装组合接口方法，
把多个有依赖关系的接口调用封装成一个方法。”

8.3 常见问题回答策略

Q: 你觉得自动化测试最大的缺陷是什么？

答：

  1. 维护成本高：UI变更会导致用例失效，需要持续维护
  2. 无法完全替代手工测试：探索性测试、用户体验测试仍需人工
  3. 初期投入大：框架搭建、用例编写需要时间，短期收益不明显
  4. 对测试人员技术要求高：需要编程能力和框架设计能力

补充：自动化测试最适合回归测试和稳定的业务流程，
新功能或频繁变更的功能不适合立即自动化。

Q: 如何把自动化测试在公司中实施并推广？

答：

  1. 选择合适的试点项目：选择业务稳定、回归频繁的项目
  2. 从小做起，快速见效：先覆盖核心流程，展示自动化价值
  3. 培训与文档：编写详细的使用文档，组织培训分享会
  4. 集成到CI/CD：将自动化测试集成到Jenkins流水线，强制卡点
  5. 持续优化：定期review用例，删除无效用例，提高执行效率
  6. 建立规范：制定自动化测试规范，代码审查标准

Q: 如何处理测试中的验证码？

答：

  1. 测试环境绕过：让开发提供万能验证码或测试账号（推荐）
  2. 数据库查询：如果验证码存入数据库，直接查询获取
  3. 图像识别：使用OCR或打码平台（成本高，不稳定）
  4. Cookie绕过：登录一次保存Cookie，后续用Cookie保持登录状态
  5. Mock：在测试环境Mock验证码验证逻辑

优先推荐：与开发协商，在测试环境提供绕过机制。
