1. PostgreSQL 简介

PostgreSQL 是一款功能强大的开源对象-关系型数据库系统，拥有超过 30 年的开发历史，以可靠性、数据完整性和标准兼容著称。它支持：

  • 丰富的数据类型（数组、JSON、范围、几何等）
  • 完整的 ACID 事务与多版本并发控制 (MVCC)
  • 函数、存储过程（多种语言）、触发器
  • 表继承、分区表、全文搜索
  • 强大的扩展能力（如 PostGIS、Citus）

4. PostgreSQL vs MySQL

维度 PostgreSQL MySQL
SQL 标准兼容 高度兼容，支持窗口函数、CTE、CHECK约束等 部分兼容，早期版本缺窗口函数等，8.0后改进
数据类型 丰富：数组、JSONB、UUID、范围、几何等 支持 JSON，但不支持数组列；数据类型较简单
自增主键 SERIAL / IDENTITY 列 AUTO_INCREMENT
存储引擎 单一引擎，MVCC 全引擎支持 多种引擎（InnoDB, MyISAM），InnoDB 为主要事务性引擎
事务隔离 支持完整的隔离级别，默认 Read Committed 支持（InnoDB），默认可重复读
复制与集群 流复制、逻辑复制、Citus 分布式；需要扩展实现读写分离 异步/半同步复制，MySQL Group Replication，InnoDB Cluster
全文搜索 内置强大全文搜索（tsvector） 内置但功能较弱（InnoDB 全文索引）
JSON 支持 JSONB（二进制，可索引）性能优异 JSON 函数和生成列索引，较方便但深度不如 JSONB
性能 复杂查询、分析查询优势；写并发可能稍弱 简单读写、高并发 OLTP 表现优秀
扩展性 扩展生态极强（PostGIS, Citus, TimescaleDB） 插件式存储引擎，但功能扩展不如 PG 灵活
易用性 初期学习曲线稍高，管理相对复杂 易安装配置，社区生态大，新手友好
优点 数据完整性严格，功能丰富，扩展强，适合复杂业务 简单快捷，流行度高，高并发读写性能好
缺点 内存消耗较大，复制配置稍复杂，插件过多需甄别 标准兼容弱，严格约束默认不开启（如SQL_MODE）
适用场景 复杂查询、地理信息、金融、数据分析、需要高级数据类型 Web应用、SaaS、标准化 LAMP/LEMP 架构

5. PostgreSQL vs Oracle

维度 PostgreSQL Oracle
许可证 开源 (MIT/PostgreSQL 许可)，免费 商业专有，许可证费用极高
SQL 语法 遵循 SQL 标准，PL/pgSQL 类似 PL/SQL PL/SQL 功能强大，语法略有不同
高级特性 分区表、物化视图、并行查询、JIT 编译、逻辑复制 分区、物化视图、RAC、Data Guard、高级队列、闪回
并发控制 MVCC，无回滚段，需要 VACUUM 清理 回滚段 + MVCC，读一致性优秀，但可能 ORA-01555
性能优化 基于统计的优化器，需要手动调参和索引 成熟的优化器和自动调优（AWR, ADDM）
高可用 流复制、Patroni 自动化切换 Data Guard (物理/逻辑), RAC（共享磁盘集群）
管理工具 pgAdmin, psql, DBeaver 等 Oracle Enterprise Manager, SQL*Plus, SQL Developer
运维成本 开源，免费社区支持或商业支持（EDB, Crunchy） 需购买许可，DBA 成本高
迁移难度 从 Oracle 迁移需注意 PL/SQL 差异，工具 ora2pg 锁定客户，迁移困难
优点 零成本，活跃社区，持续创新，可与各种平台集成 企业级最成熟，商业支持强，适合超大型混合负载
缺点 管理不如 Oracle 自动化，某些企业审计功能需插件 成本过高，厂商锁定，闭源
适用场景 中大型企业，预算敏感，需要现代数据库功能 财力雄厚的超大型企业、政府，以及要求 Oracle 特定生态

对比总结：PostgreSQL 在功能上已能与 Oracle 一较高下，且在开源生态上占有巨大优势。对于大多数新项目，如果无需 Oracle 独有的 RAC 集群、高级队列或特定法规认证，PostgreSQL 完全可胜任，且能节省大量许可费用。

6. 课后练习

请使用之前搭建的 Docker Compose PostgreSQL 环境完成以下练习。所有 SQL 语句需在 tutorial 数据库内执行。

练习1：环境与基础建表

  1. 使用 psql 或 Adminer 连接到 PostgreSQL 容器。
  2. 创建表 authors：
    • id：主键，自增整数
    • name：文本，非空
    • bio：文本，允许为空
  3. 创建表 books：
    • id：主键，自增整数
    • title：变长字符串（最长300），非空
    • author_id：整数，引用 authors(id)，外键
    • genre：文本数组
    • published_date：日期
    • metadata：JSONB 类型（存放评分、页数等）
    • price：numeric(8,2)，必须大于0
  4. 在 books.title 上创建唯一索引，在 books.metadata 上创建 GIN 索引。

练习2：数据操作与查询

  1. 向 authors 表中至少插入3位作者。
  2. 向 books 表插入至少5本书，确保使用数组字段和 JSONB 字段。例如：
    • genre 包含 {'fiction', 'adventure'}，metadata 包含 {"rating": 9.2, "pages": 320}。
  3. 写一条 SELECT 查询，列出所有书籍及其作者姓名。
  4. 写一条查询，找出所有 genre 数组中含有 fiction 的书籍。
  5. 写一条查询，从 metadata 中提取 rating，筛选评分大于 8.5 的书籍。
  6. 写一条使用窗口函数的查询，按价格降序排名，列出书名、价格和排名。

练习3：索引与性能

  1. 使用 EXPLAIN 分析前一个查询（按价格排序并排名）的执行计划。
  2. 假设有一张巨大的 books 表，经常需要按 published_date 范围查询，请写出对应的索引创建语句。
  3. 创建一个部分索引，仅索引价格大于 100 的书籍。

练习4：事务与并发

  1. 编写一个事务：先添加一本新书，然后更新对应作者的 bio 字段（如果为空则设置一个默认介绍）。若任何一步失败，全部回滚。
  2. 开启两个并行会话 (psql)，模拟并发场景：其中一个会话减少某本书的价格20%，在提交前让另一个会话读取。记录读取到的值，验证默认隔离级别下的行为。

练习5：综合小型项目

设计一个简单的“订单系统”：

  • 表 customers：id, name, email（唯一）
  • 表 orders：id, customer_id, order_date, total_amount
  • 表 order_items：id, order_id, book_id, quantity
  1. 写出对应的 CREATE TABLE 语句，确保外键关系和合适的数据类型。
  2. 插入一些示例数据。
  3. 写一个查询：列出每个客户的姓名、邮箱以及他们下过的订单总数。
  4. 写一个视图 order_detail_view，包含订单号、客户姓名、书名、数量、单价、小计（price*quantity）。
  5. 写一个函数 place_order，接收客户 ID，一个包含 book_id 和 quantity 的数组（可以使用 JSON 数组），在单个事务内创建订单和订单项，并更新书籍库存（注意：在 books 表上加 stock 列，默认10）。函数应返回新订单的 ID。
