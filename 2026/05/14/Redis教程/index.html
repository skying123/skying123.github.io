1. Redis 简介

Redis（Remote Dictionary Server）是一个开源的、基于内存的键值存储系统，通常用作数据库、缓存和消息代理。它支持多种数据结构，如字符串、哈希、列表、集合、有序集合等，并提供了高可用、持久化和集群方案。

1. 持久化机制

Redis 数据在内存中，持久化可将数据保存到磁盘，防止重启丢失。

1.1 RDB（Redis Database）

原理：在指定时间间隔内，将内存中的数据集快照写入磁盘。默认生成 dump.rdb 文件。

触发方式：

  • 自动触发：配置 save <seconds> <changes>，如 save 900 1（900秒内至少1次修改）
  • 手动触发：SAVE（阻塞） / BGSAVE（后台异步）
  • 主从全量同步时主节点自动生成 RDB

配置示例（redis.conf 片段）：

优点：文件紧凑，恢复速度快，适合冷备。
缺点：可能丢失最后一次快照后的数据。

1.2 AOF（Append Only File）

原理：以独立日志的方式记录每次写命令，重启时重新执行 AOF 文件中的命令来恢复数据。

配置示例：

AOF 重写：当文件过大时，Redis 会基于当前数据生成最小命令集的新 AOF 文件，减小体积。

  • 手动触发：BGREWRITEAOF
  • 自动触发：满足上面两个百分比和最小值条件

优点：数据安全性高，最多丢失 1 秒数据（everysec）。
缺点：文件体积大，恢复速度比 RDB 慢。

1.3 混合持久化（Redis 4.0+）

结合 RDB 和 AOF 优点：AOF 重写时将当前数据以 RDB 格式写入 AOF 文件开头，后续增量以 AOF 格式追加。
配置：aof-use-rdb-preamble yes

2. 主从复制（Replication）

2.1 原理

  • 主节点（Master）接收写请求，并将数据变更同步到从节点（Slave）。
  • 从节点可处理读请求，实现读写分离和负载均衡。
  • 同步流程：
    1. 从节点发送 PSYNC 命令。
    2. 主节点执行 BGSAVE 生成 RDB，并发送给从节点。
    3. 主节点将期间缓冲区内的写命令发送给从节点。
    4. 之后增量同步。

3. 哨兵（Sentinel）

3.1 原理

Sentinel 是 Redis 高可用方案，它监控主从节点，在主节点宕机时自动进行故障转移（Failover），将一个从节点提升为新主节点，并通知客户端新主地址。

核心功能：

  • 监控：定期 PING 节点，判断存活。
  • 通知：通知管理员或客户端故障。
  • 自动故障转移：主节点失效时选举新主。
  • 配置提供者：客户端可向 Sentinel 查询当前主节点地址。

3.2 配置哨兵

创建 sentinel.conf：

  • mymaster：主节点别名。
  • 192.168.1.100 6379：主节点 IP 和端口。
  • 2：至少需要多少个哨兵同意故障。
  • down-after-milliseconds：节点多少毫秒无响应判定为主观下线。
  • failover-timeout：故障转移超时时间。

4. 集群（Cluster）

4.1 原理

Redis Cluster 提供数据自动分片（Sharding），将数据分布在多个节点上，每个节点只保存部分数据。特性：

  • 无中心架构，各节点通过 Gossip 协议交换信息。
  • 数据按槽（slot）划分，共 16384 个槽，每个节点负责一部分槽。
  • 键属于哪个槽：CRC16(key) % 16384。
  • 支持主从复制，每个分片的主节点可有一个或多个从节点，高可用由集群自动完成（不需要额外哨兵）。
  • 客户端可连接任一节点，节点会将请求重定向到正确节点。

4.2 最低配置

至少需要 3 个主节点，生产环境建议 3 主 3 从，共 6 个节点。

5. 课后练习与答案

练习1：持久化恢复测试

  1. 使用 AOF 持久化模式启动 Redis，执行 SET test "persistence"。
  2. 重启 Redis 容器，检查 key 是否还存在。
  3. 删除 AOF 文件，再次重启会发生什么？如何恢复？

答案提示：

  • 重启后 GET test 应返回 "persistence"。
  • 删除 AOF 文件重启后数据丢失；如果有 RDB 文件并存，根据配置加载顺序可能有部分数据。可先执行 BGREWRITEAOF 确保文件最新。

练习2：主从复制数据同步

搭建一主一从环境，在主节点写入 5 个不同 key，在从节点确认数据存在，然后关闭主节点，观察从节点状态（INFO replication），说明为什么从节点不能自动成为主节点。

答案：

  • 从节点状态显示 master_link_status:down，角色仍为 slave。因为没有哨兵或手动 REPLICAOF NO ONE 提升，从节点不会自动接管。

练习3：哨兵故障转移

基于一主两从三哨兵环境，手动停止主节点，观察哨兵日志，记录故障转移时间，并通过 SENTINEL get-master-addr-by-name mymaster 获取新主地址。重新启动旧主节点后，它的角色是什么？

答案：

  • 日志会出现 +switch-master。
  • 恢复后的旧主节点会自动变为新主的从节点（配置被哨兵重写）。

练习4：集群数据分布

在 3 主 3 从集群中，写入 100 个键（如 user:1 到 user:100），然后使用 CLUSTER KEYSLOT user:50 查看槽，在对应节点上执行 DBSIZE，验证数据是否只在该节点上存在。关闭一个主节点，观察对应的从节点是否自动提升为主（通过 CLUSTER NODES 查看角色变化）。

答案：

  • CLUSTER KEYSLOT 返回槽号，数据只存于负责该槽的节点。
  • 主节点宕机后，其从节点在 cluster-node-timeout 后自动 failover，成为新主节点，CLUSTER NODES 中角色由 slave 变为 master。

6. 总结

掌握持久化让你不惧数据丢失，主从复制提供读写分离，哨兵实现高可用自动容灾，集群支撑海量数据水平扩展。建议结合 Docker 环境深入实践每个架构的部署和故障演练。
